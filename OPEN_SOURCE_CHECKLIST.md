# 开源发布检查清单

更新时间：2026-04-09

这份清单的目标不是继续扩功能，而是把项目从“本地可演示”推进到“别人能部署、能使用、能排障、能协作贡献”的开源 alpha/beta 水位。

## 发布标准

### Alpha

- 默认部署后，官网来源可以独立运行。
- 前后端出现故障时，不会静默回退到误导性假数据。
- 维护入口、配置、权限、数据库初始化路径清晰可复现。
- README、LICENSE、DEPLOY、SECURITY、CONTRIBUTING 齐备。

### Beta

- 关键 API、聚合逻辑、前端状态具备基础自动化测试。
- CI 能阻止明显回归进入主分支。
- 来源状态、同步状态、错误原因可见。
- 公众号链路被清晰标为可选增强能力，并有稳定接入流程。

### 1.0

- P0、P1 全部完成。
- P2 至少完成与维护体验和部署可靠性直接相关的部分。
- 有明确的版本发布和升级说明机制。

## P0 必补

- [x] `OS-P0-01` 拆分运行模式：把 `活动 API`、`公众号授权`、`同步 worker`、`内容补抓 worker` 改成显式启动命令。
  验收：启动 API 时不会隐式启动任何后台抓取线程。

- [x] `OS-P0-02` 收紧默认安全配置：`maintenance.allow_local_unauthenticated` 默认改为关闭。
  验收：默认配置下所有维护接口未登录返回 `401`。

- [x] `OS-P0-03` 明确公开接口与维护接口边界。
  验收：学生端接口和后台维护接口在文档和代码里都有一致的权限规则。

- [x] `OS-P0-04` 去掉生产默认 mock 兜底。
  验收：后端失败时前端显示真实故障状态和最近同步时间，不再伪装成有真实活动数据。

- [x] `OS-P0-05` 给活动聚合结果补数据新鲜度信息。
  验收：`/api/v1/wx/activities` 返回 `generated_at`、`last_success_sync_at`、`freshness`、`source_status`。

- [x] `OS-P0-06` 建立来源级失败可见性。
  验收：能看到每个来源最近成功时间、最近失败时间、失败原因、连续失败次数。

- [x] `OS-P0-07` 固化数据库初始化与管理员初始化流程。
  验收：在全新机器上按文档执行可完成 DB 初始化、管理员准备、静态资源构建和服务启动。

- [x] `OS-P0-08` 把公众号链路降级为可选增强能力。
  验收：不开微信登录时官网来源仍能提供可用活动列表，开微信登录时公众号文章能自然并入聚合。

- [x] `OS-P0-09` 在仓库根目录补统一许可证。
  验收：根目录存在单一 `LICENSE`，并写清子目录许可证关系与继承边界。

- [x] `OS-P0-10` 补 `SECURITY.md`。
  验收：说明敏感配置、默认暴露面、token/cookie 风险、漏洞提交流程。

- [x] `OS-P0-11` 补 `DEPLOY.md`。
  验收：提供单机最小部署路径、可选 worker 部署路径、环境变量说明、数据库说明、常见故障排查。

- [x] `OS-P0-12` 补 `CONTRIBUTING.md`。
  验收：明确本地开发、测试、代码风格、PR 规范、来源适配提交流程。

- [x] `OS-P0-13` 新增健康检查接口。
  验收：至少有 `liveness`、`readiness`、`source summary` 三类结果。

- [x] `OS-P0-14` 在 README 明确实验性质与合规边界。
  验收：写清公众号采集依赖登录、可能受平台策略影响、不保证长期稳定。

## P1 必补

- [x] `OS-P1-01` 给 `core/zju_activity.py` 补单元测试。
  验收：来源构建、日期抽取、详情补全、去重、排序、本科生视角排序至少有覆盖。

- [x] `OS-P1-02` 给活动 API 补集成测试。
  验收：`/activities`、`/activities/{id}`、`/colleges`、`/auth/session`、`/mps` 至少有 smoke coverage。

- [x] `OS-P1-03` 给前端补 smoke test。
  验收：覆盖接口正常、接口失败、无未来活动三种状态。

- [x] `OS-P1-04` 建 CI。
  验收：PR 自动跑后端静态校验、前端 lint/build、后端测试。

- [x] `OS-P1-05` 抽离来源适配器层。
  验收：官网抓取逻辑和来源特例分开，避免继续把全部解析逻辑堆在一个大文件里。

- [x] `OS-P1-06` 建来源注册机制。
  验收：新增一个来源时不需要修改主聚合流程，只需补配置和适配器。

- [x] `OS-P1-07` 给同步任务状态提供正式接口。
  验收：能看到队列长度、当前任务、最近历史、失败任务摘要。

- [x] `OS-P1-08` 给公众号列表返回同步统计。
  验收：每个公众号返回 `article_count`、`last_article_publish_time`、`sync_time`、`status`。

- [x] `OS-P1-09` 给官网来源提供来源级统计。
  验收：能按来源看到活动数、未来活动数、信息完整度。

- [x] `OS-P1-10` 给活动 schema 增加校验层。
  验收：聚合出的活动对象有固定字段契约，异常数据会被标记或剔除。

- [x] `OS-P1-11` 收敛配置系统职责。
  验收：`config.json`、`config.yaml`、环境变量的用途边界清晰，不再相互隐式覆盖。

- [x] `OS-P1-12` 标准化日志。
  验收：按来源 ID、任务 ID、公众号 ID 可以检索关键错误。

- [x] `OS-P1-13` 给错误做分级。
  验收：可区分网络错误、解析错误、登录错误、数据缺失错误。

- [x] `OS-P1-14` 补 Docker 生产说明。
  验收：提供单容器模式与 API + worker 模式的运行说明。

- [x] `OS-P1-15` 补 `.env.example`。
  验收：所有关键环境变量都有用途说明和默认建议。

- [x] `OS-P1-16` 在 README 提供支持矩阵。
  验收：明确哪些来源默认可用、哪些来源实验中、哪些来源依赖扫码授权。

- [x] `OS-P1-17` 增加 issue 模板。
  验收：来源失效、部署失败、功能建议、文档问题至少分开收集。

## P2 建议补

- [x] `OS-P2-01` 前端把“演示数据提示”升级成“系统状态提示”。
  验收：显示来源可用率、最近同步时间、当前是否运行在降级模式。

- [ ] `OS-P2-02` 增加管理员维护面板入口。
  验收：能在前端查看来源状态、同步状态、二维码状态，但不暴露给普通用户。

- [x] `OS-P2-03` 活动详情增加字段完整度和来源置信度提示。
  验收：用户能区分“完整活动”和“仅标题级活动”。

- [ ] `OS-P2-04` 增加抓取调试快照导出能力。
  验收：能导出原始 HTML、解析结果、错误日志，便于复现来源失效。

- [x] `OS-P2-05` 增加按来源手动刷新接口。
  验收：支持只刷新单个来源，而不是全量 refresh。

- [ ] `OS-P2-06` 引入来源 fixture 回放测试。
  验收：保存关键 HTML 样本，来源结构变化时可以快速做回归验证。

- [x] `OS-P2-07` 补 PostgreSQL 官方部署路径。
  验收：提供 SQLite 之外的推荐生产数据库方案。

- [ ] `OS-P2-08` 增加基础产品埋点或本地事件统计。
  验收：能看到用户更多在查看哪些时间段和哪些来源。

- [ ] `OS-P2-09` 增加活动屏蔽或下线机制。
  验收：误抓数据可以快速标记为不展示。

- [x] `OS-P2-10` 建版本发布说明流程。
  验收：每个 release 有迁移说明、配置变化说明、已知问题列表。

- [x] 完成剩余 P1。
- [x] 选择性完成 P2 里与可观测性和部署体验最相关的项。

## 完成判定

- [ ] 仓库在没有口头说明的情况下，外部开发者能在一台干净机器上跑起来。
- [ ] 维护者能在 10 分钟内定位“为什么今天没有活动”。
- [ ] 外部贡献者能独立新增一个来源并提交 PR。
- [x] 普通用户不会因为系统故障而看到被误认为真实数据的 mock 内容。
- [x] 即使公众号链路失效，官网来源主链路仍可继续提供服务。
