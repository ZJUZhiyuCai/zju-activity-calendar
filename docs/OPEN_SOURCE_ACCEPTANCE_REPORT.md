# Open Source Acceptance Report

更新时间：2026-04-09

## 结论

当前仓库已经达到“可以公开给别人拉起并评估”的开源 alpha 水位。

更准确地说：

- 可以作为 `v0.1.0-alpha` 候选版本进入公开发布流程。
- 不建议把当前状态表述成 beta，更不应表述成稳定可依赖 SaaS。
- 当前最合理的对外口径是：官网来源主链路可用，公众号链路是可选增强能力。

## 已通过项

### 产品与运行边界

- 根路径学生端、活动 API、来源状态、健康检查、队列状态、配置摘要已经收敛为统一主链路。
- 生产环境默认关闭前端演示数据兜底，系统故障时不会再伪装成真实活动数据。
- 官网来源聚合可以在不接公众号链路的情况下独立提供活动列表。
- 维护接口、管理员初始化、运行模式和 Docker 入口已经形成统一模型。

### 自动化验证

2026-04-09 已完成以下验证：

- 后端：`python3 -m unittest discover -s tests`，通过，当前 25 个测试。
- 前端：`npm run test:smoke`，通过。
- 前端：`npm run lint`，通过。
- 前端：`npm run build`，通过。

### 真实运行验证

- `python3 main.py -config config.yaml --mode api` 已验证 CLI 参数链路与应用启动链路可用。
- `python3 main.py -config config.yaml --mode auth` 已验证授权服务路径会进入正确运行分支。
- 当前沙箱环境限制了端口绑定与部分外网访问，因此这里无法完成完整的公网运行验收。

## 本轮收口内容

- 修复了运行时 CLI 与文档不一致的问题：`main.py` 现在正式接受 `-config/--config`。
- 收敛了公众号授权定时刷新开关，避免 `auth/all` 模式继续隐式依赖旧环境变量。
- 为新的运行时标志补充了测试覆盖。
- 移除了遗留的 `we-mp-rss/compose/test.yaml`，避免旧镜像名和废弃环境变量继续误导维护者。
- 更新了 `OPEN_SOURCE_CHECKLIST.md` 的阶段状态与验收快照，使文档与当前事实一致。

## 未签收项

下面这些不是“代码明显缺失”，而是“还没有完成严肃验收闭环”：

### 1. 干净机器部署演练

尚未在一台没有口头说明、没有本地历史状态的干净机器上完成从零部署验收。

这项没做之前，不能把“外部开发者可独立部署”当作已完全签收。

### 2. 维护者排障演练

尚未做一次带时间约束的真实排障演练，例如：

- 模拟今天没有活动
- 要求维护者在 10 分钟内判断是来源失效、worker 未运行、授权失效，还是配置错误

当前接口和日志大概率已经够用，但这项还没有被真实证明。

### 3. 外部贡献者新增来源演练

来源适配器层、来源注册机制和文档都在，但还没有做一次“陌生贡献者独立新增来源并提交 PR”的完整演练。

这意味着“别人能协作贡献”在机制上基本成立，在流程上还没被实战签收。

## 已知风险

### P1/P2 之外的现实风险

- 公众号链路受扫码授权、平台策略、网络出口和上游页面结构影响，不应承诺稳定性。
- 当前环境对 `mp.weixin.qq.com` 的访问和解析能力强依赖部署网络条件。
- 当前没有管理员维护面板，维护能力主要仍依赖接口、日志和手工排障。

### 尚未完成的 P2

- 管理员维护面板入口
- 抓取调试快照导出
- 来源 fixture 回放测试
- 基础产品埋点或本地事件统计
- 活动屏蔽或下线机制

这些项不会阻止 alpha 发布，但会限制维护效率和后续演进速度。

## 发布建议

建议按下面的严格口径推进：

### 可以现在做的

- 以 `v0.1.0-alpha` 为目标整理公开仓库。
- 按 `README.md`、`DEPLOY.md`、`RELEASING.md`、`docs/releases/v0.1.0-alpha.md` 继续准备发布材料。
- 对外明确：主链路是官网来源聚合，公众号链路是可选增强能力。

### 发布前最好补的一次人工验收

- 在一台干净机器上按文档做一次完整冷启动部署。
- 手工访问根路径、活动接口、健康检查接口。
- 手工确认前端构建缺失时会返回真实 `503 frontend_not_built`，而不是假数据。

### 当前不建议宣称的能力

- 不建议宣称“稳定生产可用”。
- 不建议宣称“公众号链路长期可靠”。
- 不建议宣称“外部贡献者已经零门槛可上手”。

## 最终判断

结论是：可以进入公开 alpha 发布准备，但还不应把自己骗成 beta。

如果你现在要把项目发出去，我的严肃判断是“可以发 alpha，前提是发布说明把边界写清楚”；如果你要把它当成“别人拿去就能稳定用”的项目，那还差最后一轮真实环境验收，而不是继续堆功能。
