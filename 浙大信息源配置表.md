## 一、来源优先级原则

从本科生视角看，来源优先级按以下顺序判断：

1. 是否能提供未来活动
2. 是否能稳定给出时间和地点
3. 是否对本科生公开、通用、低门槛
4. 是否能识别校区，便于判断顺不顺路
5. 是否更新频繁

结论：
- 校级公开讲座平台优先级最高
- 活动频繁且信息完整的学院优先级高
- 只有历史存量、没有近期活动的来源优先级下降
- 解析失败但理论价值高的来源，需要进入优先修复队列
- 能提前发出未来讲座预告的公众号，优先级应高于仅有历史归档的官网页面

## 三、公众号来源策略

公众号主要解决的不是“补数量”，而是“补未来活动预告”。

### 3.1 优先接入的讲座类公众号

| 类型 | 公众号 | 作用 |
|------|--------|------|
| 校级 | 浙江大学图书馆 | 公开讲座、培训、讲座海报 |
| 校级 | 浙江大学本科生院 | 本科生活动预告、讲座通知 |
| 校级 | 浙江大学研究生院 | 校级讲座和论坛补充 |
| 学院 | 浙江大学管理学院 | 已验证高频，适合补未来活动 |
| 学院 | 浙江大学经济学院 | 讲座和论坛类活动补充 |
| 学院 | 浙江大学文学院 | 人文讲座未来预告 |
| 学院 | 浙大计算机和软件学院高级培训 | 工科类讲座与培训 |
| 学院 | 浙江大学医学院 | 医学讲座补充 |

### 3.2 公众号接入原则

- 先接“讲座/论坛/报告/训练营/活动预告”高信号公众号
- 先复用 `we-mp-rss` 已同步的 `Feed/Article` 数据，不重新造轮子
- 没有明确活动日期的公众号文章，不直接进入日历主链路
- 公众号文章优先用于补“未来活动”，官网优先用于补“稳定结构化信息”
- 当前本地数据库里公众号 `feeds/articles` 仍为 0，接下来需要先完成接入与抓取，公众号链路才会产出真实数据

### 3.3 当前执行状态与前提

- 公众号文章到活动结构的映射代码已经接通，当前不是“还没支持公众号”，而是“还没完成真实接入和同步”
- 授权二维码接口已经修复为稳定静态路径 `/static/wx_qrcode.png`
- 浏览器授权链路已补上：
  - 绝对二维码文件路径
  - 浏览器自动回退
  - 扫码场景允许加载图片
- 同时已补通 API 授权链路：
  - 适配新版微信登录流程 `startlogin -> scanloginqrcode?action=getqrcode`
  - `WERSS_AUTH_WEB=False` 时可直接生成本地二维码
  - 本地已验证 `/api/v1/wx/auth/qr/code` 返回 `is_exists=true`
- 因此公众号接入的现实前置动作已经变成：
  - 用 `WERSS_AUTH_WEB=False` 启动 `we-mp-rss`
  - 扫描 `http://127.0.0.1:8006/static/wx_qrcode.png`
  - 扫码后优先把图书馆、本科生院、研究生院、管理学院、经济学院、文学院纳入采集名单

这里要补一个产品层面的约束：

- 这些动作是后台采集动作，不应成为学生端主界面
- 学生端只应该看到已经整理好的活动日历与预览结果
- 因此信息源接入的成功标准，不是“能进后台加公众号”，而是“首页未来活动变多、变清楚”

## 五、配置维护规则

### 4.1 `config.json` 中一个来源至少应具备
- `id`
- `name`
- `url`
- `priority`
- `category`

对实际要接入抓取的学院来源，还应补：
- `lecture_url`
- `parser_type`
- 如果默认解析不稳定，补充 `selectors`

### 4.2 新来源接入标准
只有满足以下至少两条，才值得优先接入：
- 最近 30 天内有更新
- 标题能看出讲座/论坛/活动属性
- 列表页能稳定取到日期
- 详情页能较稳定取到时间/地点/主讲人
- 对本科生存在明显价值

### 4.3 降级标准
以下来源应降低优先级：
- 只有历史数据，没有近期更新
- 页面长期为空
- 结构频繁变化导致维护成本过高
- 活动面向研究生或行政通知为主，对本科生日常决策帮助有限

## 六、本科生视角下的来源标签

为后续排序和预览做准备，来源可按以下标签理解：

| 标签 | 说明 | 示例 |
|------|------|------|
| `core_open` | 校级公开入口，适合广泛展示 | 图书馆讲座、本科生院 |
| `college_high_signal` | 活动质量高且字段完整 | 管理学院、文学院 |
| `college_archive` | 历史活动多于近期活动 | 计算机学院 |
| `broken_source` | 理论价值存在，但当前不可用 | 本科生院、研究生院、光电学院 |
| `low_decision_value` | 虽有内容，但对快速决策帮助有限 | 只有标题和日期的来源 |
| `wechat_future_signal` | 公众号里更早出现的未来活动预告 | 本科生院公众号、图书馆公众号 |

这些标签不一定要写回 `config.json`，但后续后端可基于来源 ID 推导相应的预览权重。

## 七、当前结论

截至 2026-04-07：
- 真正支撑“本科生快速预览”的来源还不够多
- `som` 是当前最像真实产品入口的学院来源
- `library` 和 `undergraduate_school` 仍然是最值得继续投入的校级来源
- 当前最大的来源问题不是“数量太少”，而是“未来活动太少、稳定来源太少”
- 公众号是未来活动预告的重要来源，不能只作为后置增强
- 公众号聚合已经进入可执行状态，但它的价值体现在首页日历，而不是采集维护后台本身

因此，接下来不建议继续无差别扩来源，而应按下面顺序推进：

1. 先让学生端首页稳定展示这些来源产出的活动
2. 修高价值但当前失效的校级来源
3. 接入讲座类公众号，优先补未来活动
4. 稳住已能产出未来活动的来源
5. 再扩新的学院来源

## 八、与产品目标的关系

这份配置表服务于同一个目标：

**不是做“浙大活动采集大全”，而是做“本科生今天就能用来挑活动的预览产品”。**

所以信息源的衡量标准最终只有一个：

**它能不能帮助用户更快决定“这场我要不要去”。**
