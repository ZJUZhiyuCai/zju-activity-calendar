# 浙大活动日历技术方案

更新时间：2026-04-07

## 一、北极星目标

### 1.1 产品目标
做一个面向浙大本科生的活动预览工具，不追求先把全校所有活动都收全，而是先解决一个更具体的问题：

**让本科生在 30 秒内看懂“这周有什么值得去、离我近不近、信息够不够完整、要不要点开”。**

### 1.2 核心用户
- 浙大本科生
- 日常主要在紫金港生活学习的学生
- 想顺手看讲座、论坛、训练营、分享会，而不是专门查通知的用户

### 1.3 核心任务
- 快速预览今天、明天、本周活动
- 优先看到更可能和自己相关的活动
- 先用极少的信息判断值不值得点开
- 在点开前就知道时间、地点、校区、主办方是否明确

### 1.4 不做的事
- 现阶段不做“全校活动数据库”
- 现阶段不做复杂报名工作流
- 现阶段不做重运营内容流
- 现阶段不只依赖单一来源链路

---

## 二、当前产品定义

### 2.1 当前版本定位
当前版本不是“原始活动列表聚合器”，而是“本科生快速预览界面”。

前端已经具备：
- 月历视图
- 学院/来源筛选
- 搜索
- 右侧快速预览栏
- 详情补拉
- 接口不可用或无近期数据时的前端降级回退

### 2.2 当前产品原则

#### 原则 A：预览优先于完整
同一屏内优先展示最值得先判断的活动，而不是平均展示所有活动。

#### 原则 B：近期优先于历史
对本科生来说，“这周能不能去”比“去年办过什么”更重要。

#### 原则 C：可决策优先于可抓取
如果一个来源虽然能抓到标题，但没有时间地点，对预览价值就偏低。

#### 原则 D：顺路优先
校区、时间段、是否校级公开，都是本科生是否会去的重要判断因素。

## 三、当前实现架构

### 3.1 技术栈

```text
前端：React + Vite
后端：FastAPI（基于 we-mp-rss 项目扩展）
采集：requests + BeautifulSoup
配置：config.json
缓存：进程内缓存
```

### 3.2 当前系统结构

```text
frontend/
  src/App.jsx                         页面编排与前端降级逻辑
  src/components/Calendar.jsx         月历视图
  src/components/PreviewRail.jsx      本科生快速预览
  src/components/ActivityDetail.jsx   详情弹层
  src/api/index.js                    前端 API 与辅助方法

we-mp-rss/
  core/zju_activity.py                浙大活动聚合服务
  apis/zju_activity.py                活动接口
  web.py                              API 与前端入口编排
  static/calendar/                    React 构建产物

config.json                           信息源配置
```

### 3.3 当前产品入口与数据链路

```text
浏览器打开根路径 /
  -> 直接进入 React 日历首页
  -> 同源请求 /api/v1/wx/activities /colleges
  -> 后端聚合官网来源 + 公众号文章
  -> 抽取标题 / 日期 / 时间 / 地点 / 主讲人
  -> 生成统一活动结构
  -> 前端按“本科生快速预览”方式展示
```

这里需要明确：

- `frontend/` 才是产品本体
- `we-mp-rss/` 只是内部采集与聚合层
- 公众号授权、接入、同步都属于后台运维动作，不应该成为学生端主流程
- 旧 RSS 输出和资源反代已经不再属于当前产品主链路

### 3.4 当前 API

```text
GET /api/v1/wx/activities
GET /api/v1/wx/activities/search
GET /api/v1/wx/activities/{id}
GET /api/v1/wx/colleges
```

其中 `/activities` 已支持以下面向预览的参数：
- `student_view=true`
- `sort_by=relevance`
- `upcoming_only=true`

## 四、当前真实状态

以下状态基于 2026-04-07 的本地验证。

### 4.1 数据概况
- 聚合活动总数：56
- 未来活动数：3
- 时间和地点都明确的活动数：19
- 当前主要可用来源集中在 `som`、`lit`、`library`、`international_college`
- 本地 `we-mp-rss/data/db.db` 当前 `feeds/articles` 均为 0，公众号链路已接入代码，但仍需先完成真实接入与同步

### 4.2 当前主要问题

#### 问题 1：数据新鲜度不足
当前能聚合到的历史活动不少，但真正未来可看的活动很少。对本科生来说，这会直接削弱产品价值。

#### 问题 2：官网与公众号覆盖面不同
官网更适合作为稳定保底来源，但未来活动预告、临时加场、报名提醒，往往更早出现在公众号。

#### 问题 3：来源可用率仍然决定体验上限
后端已经补上预览元数据，但如果近期真实来源仍然过少，前端体验仍会更多依赖降级策略。

#### 问题 4：信息完整度不一致
部分来源只有标题和日期，缺少时间地点；这类活动对“快速决策”帮助有限。

### 4.3 公众号接入执行状态
- `we-mp-rss` 已经能直接读取本地公众号数据库，并把文章映射成统一活动结构
- 本地已验证公众号文章可抽取出日期、时间、地点、校区、报名链接，并进入活动聚合结果
- 浏览器授权链路已补上三项关键修复：
  - 登录二维码文件改为绝对路径，避免依赖启动目录
  - 登录浏览器支持 `firefox -> chromium -> webkit` 自动回退
  - 扫码登录场景不再主动禁用图片资源，避免二维码被拦截
- 在当前机器上，API 授权链路已经打通：
  - `wx_api.py` 已切到新版登录流程：`startlogin -> scanloginqrcode?action=getqrcode`
  - `WERSS_AUTH_WEB=False` 时，可直接生成 `/static/wx_qrcode.png`
  - 本地已验证 `GET /api/v1/wx/auth/qr/code` 返回 `is_exists=true`
- 因此当前真正可执行的路径不是继续卡在 Playwright，而是直接走 API 授权模式

### 4.4 当前阶段的执行顺序
当前顺序不能再是“先看到后台、再想办法进入日历”，而应该改成：

1. 先把根路径稳定落到学生端日历首页
2. 保证日历首页在无数据、少数据、有真实数据三种状态下都能清楚展示
3. 再把公众号授权与同步当作后台采集动作补上
4. 最后继续扩高价值来源，提升未来讲座密度

当前本地后台能力仍然可用：
- 服务地址：`http://127.0.0.1:8006`
- 二维码图片：`http://127.0.0.1:8006/static/wx_qrcode.png`

但这条链路的定位已经变成：

- 供维护者完成公众号采集授权
- 供后端补未来活动数据
- 不直接暴露为学生端产品首页

## 五、本科生预览模型

### 5.1 预览排序的判断顺序
在本科生视角下，一个活动是否应该被优先展示，优先看以下信号：

1. 是否即将发生
2. 是否有明确时间和地点
3. 是否属于校级公开入口或高频公共来源
4. 是否能识别校区
5. 是否给出主讲人
6. 是否明显偏研究生场景

### 5.2 预览元数据定义
后端现已直接输出以下字段，前端优先消费这些字段，减少重复判断：

| 字段 | 含义 | 用途 |
|------|------|------|
| `campus` | 从地点提取的校区 | 判断是否顺路 |
| `is_upcoming` | 是否未来活动 | 历史活动降权 |
| `days_until` | 距今天还有几天 | 今天/明天/本周预览 |
| `has_complete_info` | 是否同时具备时间和地点 | 决策信号 |
| `preview_reason` | 适合展示给用户的短标签 | 例如“校级入口”“紫金港校区” |
| `student_score` | 本科生视角排序分数 | 后端统一排序逻辑 |

### 5.3 当前前端体验方向
右侧快速预览栏已经朝这个模型在走：
- 支持“与我相关 / 今天 / 明天 / 本周 / 已选日期”
- 强化时间、地点、校区、主办方的前置展示
- 用摘要卡而不是长列表帮助判断

这套逻辑已经沉到后端，接下来重点是让更多真实来源产出这些字段。

## 六、信息源策略

### 6.1 双主链路
现阶段不是“官网或公众号二选一”，而是双主链路：

- 官网：稳定、结构明确、适合作为保底活动源
- 公众号：更容易出现未来活动预告、讲座海报、报名提醒和临时通知

### 6.2 官网链路定位
官网适合承接：
- 学院讲座栏目
- 校级讲座平台
- 结构化列表抓取
- 详情页补充时间地点

### 6.3 公众号链路定位
公众号适合承接：
- 未来活动预告
- 尚未进入官网栏目但已经开始宣传的讲座
- 报名链接、海报、活动亮点
- 学院、图书馆、本科生院、研究生院这类高频通知源

因此，公众号不是“后置增强”，而是未来活动预览必需链路。

## 七、路线图

### 7.1 P0：保证“最近可看”
目标：用户打开时能优先看到未来活动，而不是被历史活动淹没。

需要完成：
- 根路径直接进入日历，而不是旧后台或采集工具
- 前端默认同源请求后端活动接口
- 更清晰的 upcoming 过滤策略
- 历史活动显式降权
- 无近期数据时的稳定回退
- 公众号未来活动并入聚合结果

### 7.2 P1：扩大预览元数据的真实覆盖率
目标：不是再定义字段，而是让更多真实来源能产出这些字段。

需要继续完成：
- 提升 `campus` 的识别率
- 提升时间地点抽取命中率
- 让更多校级来源产出未来活动
- 减少只有标题和日期的低信息活动
- 让公众号文章也能稳定产出活动日期、地点和主讲人

### 7.3 P2：提升真实来源可用率
目标：提高未来活动的真实占比。

优先处理：
- `graduate_school`
- `undergraduate_school`
- `graduate_calendar`
- `cec`
- `cmm`
- `cst`
- `opt`

### 7.4 P3：扩展公众号覆盖面
目标：在讲座类公众号跑顺后，再扩大到更广的活动来源。

扩展方向：
- 学园/团委/学工类活动预告
- 报名链接
- 活动封面图
- 更完整的正文摘要

## 八、当前阶段的工程结论

### 8.1 我们已经确认的结论
- 产品方向应从“活动大全”切到“本科生快速预览”
- UI 已经具备日历与预览雏形，但入口必须直达这个界面
- 当前最大瓶颈不是界面，而是“近期真实活动太少”
- 后端已经具备一层面向本科生的预览语义
- 仅靠官网不足以覆盖未来活动预告，公众号需要进入正式链路
- 公众号聚合代码已基本接通，但它应退到后台能力，而不是产品表面

### 8.2 当前最值得继续做的事
当前最值得继续做的事是三件事一起对齐：

1. 让学生打开首页时第一眼就是日历与快速预览
2. 打通讲座类公众号到活动聚合的链路
3. 提高官网与公众号两条链路的近期真实可用率

这三件事一起成立，产品才会像“真能用”的本科生活动预览工具。

## 九、当前迭代约束

### 9.1 约束
- 以 `config.json` 为信息源配置事实来源
- 前端需保留 mock 回退，方便持续调 UI
- 后端接口尽量兼容 Python 3.9 环境
- 不引入重型基础设施

### 9.2 验收标准
当一个本科生打开页面时，应能在不读长文的前提下快速回答：
- 今天有什么能去？
- 明天有什么能去？
- 本周哪个最值得先点开？
- 这个活动时间地点清楚吗？
- 这个活动在哪个校区？

如果这些问题还不能稳定回答，说明产品还没有达到当前阶段目标。
